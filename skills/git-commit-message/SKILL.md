# Git Commit 规范

所有commit消息必须遵循 `<前缀>: <中文描述>` 格式。

## 前缀说明

| 前缀 | 用途 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat: 增加用户注册功能` |
| `fix` | 修复bug | `fix: 修复页面崩溃的问题` |
| `docs` | 仅文档相关的修改 | `docs: 更新README文件` |
| `style` | 代码风格变动，不影响逻辑 | `style: 删除多余的空行` |
| `refactor` | 代码重构，不新增功能也不修复bug | `refactor: 重构用户验证逻辑` |
| `perf` | 提升性能 | `perf: 优化图片的加载速度` |
| `test` | 测试相关 | `test: 增加用户模块测试` |
| `chore` | 杂项，构建过程或辅助工具变动 | `chore: 更新依赖库` |
| `revert` | 回退 | `revert: 回滚feat: 增加用户注册功能` |
| `ci` | CI配置文件和脚本的变更 | `ci: 修改GitHub Actions配置文件` |
| `build` | 影响构建系统的更改 | `build: 升级CMake到3.10版本` |
| `merge` | Git合并 | `merge: 合并feature/xxx分支` |

## 前缀选择规则

- **feat**: 新增文件、新增函数/类/模块、新增API端点、新增页面/组件
- **fix**: 修复错误行为、解决崩溃/异常、修正逻辑错误、纠正返回值
- **docs**: 修改`.md`、`.rst`、注释等纯文档内容
- **style**: 格式化调整、去除空行/多余空格、分号/引号风格统一、不影响AST的修改
- **refactor**: 重命名变量/函数、提取公共代码、调整目录结构、简化逻辑但不改变行为
- **perf**: 算法优化、减少内存分配、缓存引入、懒加载
- **test**: 新增/修改测试用例、测试夹具、测试配置
- **chore**: 依赖更新、构建脚本调整、`.gitignore`修改、没有生产代码变化的杂务
- **revert**: 撤销之前的某次提交
- **ci**: `.github/workflows/`、`Jenkinsfile`、`.gitlab-ci.yml`等CI配置
- **build**: `CMakeLists.txt`、`Makefile`、`webpack.config.js`等构建配置
- **merge**: 合并分支产生的commit

## 描述编写要求

- **使用中文**，简洁明了
- 完整的一句话，动词开头
- 不超过50个字符（中文约25个字）
- 描述做了什么，而非怎么做
- 以动词开头：增加、修复、优化、移除、重构、更新、回滚、合并

## 执行流程

### 1. 检查变更

先执行以下命令获取变更信息：

```bash
git status
git diff --stat
git diff --cached --stat
```

如果暂存区为空，询问用户是否需要 `git add`。

### 2. 分析变更类型

根据diff内容判断变更性质，选择最匹配的前缀：

- 优先看是否有新文件/新功能 → `feat`
- 是否有bug修复 → `fix`
- 是否只是文档调整 → `docs`
- 是否只是测试代码 → `test`
- 其余按前缀选择规则匹配

### 3. 生成commit消息

根据变更内容，用中文描述核心改动，生成完整的commit消息。

### 4. 用户确认

将生成的commit消息展示给用户确认。格式如下：

```
建议的commit消息：
<前缀>: <中文描述>

涉及文件：
- file1.py
- file2.py

确认提交？(yes/no)
```

### 5. 执行提交

用户确认后执行：

```bash
git add <files>
git commit -m "<前缀>: <中文描述>"
```

如果用户拒绝，根据反馈调整commit消息后重新确认。

## 注意事项

- 一次commit尽量只做一件事，如果变更涉及多种类型，选择最主要的前缀
- 如果暂存区混合了不相关的变更，建议用户拆分为多个commit
- 描述中不需要句号结尾
- 不要包含"修改了"这种模糊描述，要具体说明修改了什么
- `perf`不是`pref`，注意拼写
- 使用英文符号, 如冒号、逗号、括号等，保持格式统一
